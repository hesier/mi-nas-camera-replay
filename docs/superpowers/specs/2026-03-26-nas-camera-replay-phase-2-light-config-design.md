# NAS 摄像头回放系统第二阶段设计方案（轻配置版）

**日期：** 2026-03-26

**状态：** 已确认，可进入实施规划

## 1. 背景与目标

第一阶段已经完成“单摄像头、单用户、本地网页访问”的基础能力，当前系统可以完成离线索引、按天回放、时间轴展示、断档提示与原始 MP4 播放。

第二阶段目标是在尽量保持配置简单、页面简单和实现简单的前提下，补充两项能力：

- 支持多摄像头
- 支持单用户登录

本阶段明确采用“轻配置版”方案：

- 多摄像头只通过 `.env` 配置，不引入摄像头管理页面
- 页面不展示自定义摄像头名称，只展示固定编号
- 登录只通过 `.env` 中的单个密码字段控制
- 不引入用户表，不支持注册、改密、多用户

## 2. 已确认范围

### 2.1 本阶段范围

- 通过多个 `VIDEO_ROOT_n` 配置多个摄像头目录
- 后端按摄像头编号分别索引、分别聚合、分别查询
- 前端支持“通道切换”
- 当仅有一个摄像头时，前端不显示通道切换器
- 增加基于单个密码的登录页
- 所有业务接口都要求登录后访问

### 2.2 本阶段不做

- 摄像头名称管理
- 摄像头增删改页面
- 用户表、注册、改密、退出其他设备
- 多用户与权限分级
- 同屏多摄像头播放
- 云端部署安全强化
- 热更新 `.env` 配置

## 3. 方案对比与结论

### 3.1 方案 A：引入 `cameras` 表与 `users` 表

优点：

- 扩展性最好

缺点：

- 超出当前“配置尽量简单”的目标
- 会引入额外的管理概念与 UI

### 3.2 方案 B：纯 `.env` 配置 + SQLite 记录 `camera_no`

优点：

- 配置简单
- 后端查询结构清晰
- 在不增加管理复杂度的前提下支持多摄像头

缺点：

- 修改目录配置后需要重启服务
- 未来扩展多用户时需要重构认证层

### 3.3 方案 C：单根目录 + 子目录自动映射摄像头

优点：

- 配置项最少

缺点：

- 强依赖 NAS 目录结构
- 灵活性不足

### 3.4 最终结论

第二阶段采用 **方案 B：纯 `.env` 配置 + SQLite 记录 `camera_no`**。

这条路线保留了“本地工具、轻配置”的特点，同时避免把多摄像头逻辑散落在查询代码中。

## 4. 配置设计

### 4.1 `.env` 格式

```env
VIDEO_ROOT_1=/nas/cam1
VIDEO_ROOT_2=/nas/cam2

APP_PASSWORD=your-password

INDEX_ON_STARTUP=false
INDEX_SCHEDULER_ENABLED=false
INDEX_SCHEDULER_TIME=03:00
SQLITE_URL=sqlite:///./replay.db
TIMEZONE=Asia/Shanghai
```

### 4.2 配置解析规则

- 只识别键名符合 `VIDEO_ROOT_数字` 的配置项
- 数字部分直接作为摄像头编号 `camera_no`
- 允许跳号，例如只配置 `VIDEO_ROOT_1` 与 `VIDEO_ROOT_3`
- 至少必须存在一个 `VIDEO_ROOT_n`
- `APP_PASSWORD` 必须非空
- 不允许两个通道配置为相同目录
- 不允许两个通道配置为父子嵌套目录
- 一旦发现目录重复或重叠，服务启动失败，并给出明确报错
- `/api/cameras` 返回的通道顺序固定为 `camera_no` 升序
- 修改 `.env` 后需要重启服务才能生效

### 4.3 运行时内部表示

后端启动后将配置解析为如下结构：

```python
[
    {"camera_no": 1, "video_root": "/nas/cam1"},
    {"camera_no": 2, "video_root": "/nas/cam2"},
]
```

系统其他模块不再直接处理 `VIDEO_ROOT_n` 字符串，而统一围绕 `camera_no` 工作。

## 5. 数据模型调整

第二阶段不新增 `cameras` 表，也不新增 `users` 表。

### 5.1 `video_files`

新增字段：

- `camera_no: Integer, nullable=False`

建议索引：

- `camera_no + actual_start_at`
- `camera_no + file_mtime`

约束建议：

- 继续保留 `file_path` 唯一约束即可
- 若后续发现路径策略变化，再考虑改为 `camera_no + file_path` 联合唯一

### 5.2 `timeline_segments`

新增字段：

- `camera_no: Integer, nullable=False`

建议索引：

- `camera_no + day + segment_start_at`

查询语义调整为：

- 一个时间轴只属于一个 `camera_no + day`

### 5.3 `day_summaries`

新增字段：

- `camera_no: Integer, nullable=False`

结构调整：

- 不能再只使用 `day` 作为唯一键
- 第二阶段允许直接重建 SQLite，不要求兼容旧库
- 最简单做法是新增自增主键 `id`，并增加 `camera_no + day` 联合唯一约束

### 5.4 数据库升级策略

本项目当前为本地工具型项目，第二阶段接受以下升级策略：

1. 修改 ORM 模型与初始化逻辑
2. 删除旧 `replay.db`
3. 重新全量索引

本阶段不引入通用迁移框架。

如果后续修改了 `VIDEO_ROOT_n` 集合，尤其是删除、改号或重指向目录，推荐同步删除 `replay.db` 后重新全量索引。第二阶段不保证“未配置但仍残留在数据库中的历史通道数据”继续可访问。

对于第一阶段遗留的旧版 `replay.db`，第二阶段采用“**显式报错，不自动删除**”策略：

- 启动时如果发现数据库结构与当前模型不兼容，服务启动失败
- 错误信息中明确提示删除 `replay.db` 后重新全量索引
- 第二阶段不自动迁移旧库，也不自动清空旧库

## 6. 索引与重建链路

### 6.1 新的索引流程

```text
读取所有 VIDEO_ROOT_n
→ 对每个 camera_no 单独扫描目录
→ 解析文件名与媒体信息
→ 写入 video_files(camera_no, ...)
→ 计算受影响日期
→ 重建该 camera_no 下这些日期的 timeline_segments
→ 更新该 camera_no 下的 day_summaries
```

### 6.2 模块职责调整

#### 文件扫描模块

职责：

- 接收 `camera_no` 与对应目录
- 返回该通道下的候选文件集合

不负责：

- 跨通道聚合

#### 索引任务模块

职责：

- 遍历所有配置出的摄像头目录
- 为每个文件写入正确的 `camera_no`
- 触发按 `camera_no + day` 的重建

#### 日期重建模块

重建接口概念上调整为：

```text
rebuild_day_timeline(session, camera_no, day)
```

而不是仅按 `day` 重建。

### 6.3 核心约束

- 所有时间轴聚合必须带上 `camera_no`
- 所有日期摘要聚合必须带上 `camera_no`
- 不允许不同通道同一天的数据出现在同一条时间轴中

## 7. API 设计

### 7.1 通道列表接口

新增：

`GET /api/cameras`

返回当前配置可用的通道列表：

```json
[
  { "cameraNo": 1, "label": "通道 1" },
  { "cameraNo": 2, "label": "通道 2" }
]
```

该接口只基于配置生成，不依赖数据库中的摄像头表。

额外约束：

- 返回结果按 `cameraNo` 升序排列
- 仅返回当前 `.env` 中已配置的通道

### 7.2 日期列表接口

调整为：

`GET /api/days?camera=1`

返回值结构保持现有格式不变，但只返回指定通道的数据。

参数规则：

- `camera` 为必填整数参数
- 当 `camera` 不在当前配置的通道集合中时，返回 `404`
- 当该通道已配置但当前没有任何日期数据时，返回 `200` 与空数组

### 7.3 时间轴接口

调整为：

`GET /api/timeline?camera=1&day=2026-03-26`

返回值结构保持现有格式不变，但只返回指定通道、指定日期的时间轴。

参数规则：

- `camera` 为必填整数参数
- 当 `camera` 不在当前配置的通道集合中时，返回 `404`
- 当该通道存在但该日期没有时间轴数据时，沿用第一阶段语义返回 `404`

### 7.4 定位接口

沿用第一阶段的 GET 风格，仅增加通道维度：

`GET /api/locate?camera=1&at=2026-03-26T12:30:00+08:00`

说明：

- `day` 不单独传递，继续由 `at` 推导
- 返回结构沿用第一阶段语义
- 匹配范围限定在指定通道内
- 当 `camera` 不在当前配置的通道集合中时，返回 `404`

### 7.5 视频流接口

`GET /api/videos/{fileId}/stream`

保持接口路径不变。

原因：

- `fileId` 已经是数据库主键
- 前端播放器复用成本最低

但该接口从第二阶段开始也需要登录校验。

## 8. 登录与认证设计

### 8.1 目标

登录仅用于保护本地工具页面与业务接口，不引入账号体系。

### 8.2 配置

`.env` 中仅新增一个认证字段：

```env
APP_PASSWORD=your-password
```

### 8.3 认证接口

新增：

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/status`

`POST /api/auth/login` 请求体：

```json
{
  "password": "your-password"
}
```

`GET /api/auth/status` 返回示例：

```json
{
  "authenticated": true
}
```

### 8.4 登录态保存方式

第二阶段使用 **HttpOnly 签名 Cookie** 保存登录态。

规则：

- 登录成功后由后端写入 Cookie
- 前端不保存明文密码
- 后续业务请求依赖 Cookie 自动鉴权
- 登出时清理 Cookie

### 8.5 Cookie 策略

- `HttpOnly`
- `SameSite=Lax`
- 本地开发环境可不强制 `Secure`
- 采用会话级 Cookie，浏览器关闭后失效
- 第二阶段不额外增加新的密钥配置项
- Cookie 签名密钥可基于 `APP_PASSWORD` 派生

这一做法的目标是减少配置项，而不是追求公网部署级别的安全隔离。

### 8.6 受保护接口范围

以下接口都要求登录：

- `/api/cameras`
- `/api/days`
- `/api/timeline`
- `/api/locate`
- `/api/videos/{fileId}/stream`
- `/api/index/rebuild`

未登录时统一返回 `401`。

## 9. 前端交互设计

### 9.1 页面结构

```text
未登录
→ LoginPage

已登录
→ ReplayPage
   → 通道切换
   → 日期选择
   → 时间轴
   → 播放器
```

### 9.2 通道展示规则

- 只有一个通道时，不显示通道切换器
- 有多个通道时，显示固定文案：
  - `通道 1`
  - `通道 2`
  - `通道 3`
- 首次进入回放页时，默认选中 `cameraNo` 最小的通道

不展示用户可编辑名称。

### 9.3 切换通道行为

切换通道时，前端必须执行以下重置：

- 清空当前播放片段
- 清空当前选中时间
- 重置当前日期
- 重新请求该通道的日期列表
- 如果日期列表非空，自动选中最新日期，再请求该日期对应时间轴
- 如果日期列表为空，停止后续时间轴请求，并显示“该通道暂无录像”空状态

### 9.4 登录页行为

- 页面仅有一个密码输入框
- 登录成功后进入回放页
- 登录失败时显示统一错误提示
- 已登录状态下刷新页面应自动恢复
- 回放页提供最小可见的“退出登录”入口
- 用户点击退出登录后调用 `/api/auth/logout` 并返回登录页

## 10. 非目标与边界

为保持方案简单，本阶段接受以下限制：

- 修改摄像头目录配置需要重启服务
- 修改密码需要重启服务
- 不支持在线改密码
- 不支持多用户
- 不支持记录登录者身份
- 不支持细粒度权限控制

这些限制是当前设计的明确边界，而不是待补缺陷。

## 11. 测试要求

### 11.1 后端测试

- 配置层能够解析多个 `VIDEO_ROOT_n`
- 缺少 `APP_PASSWORD` 时启动失败
- 同一天不同通道的数据不会混在一起
- `/api/days?camera=1` 只返回通道 1 数据
- `/api/timeline?camera=2&day=...` 只返回通道 2 数据
- 未登录访问受保护接口返回 `401`
- 正确密码登录成功
- 错误密码登录失败
- 登录后可访问视频流接口

### 11.2 前端测试

- 多通道时显示通道切换器
- 单通道时不显示通道切换器
- 切换通道会重新加载日期与时间轴
- 未登录时显示登录页
- 登录成功后进入回放页
- 退出登录后回到登录页

## 12. 实施顺序建议

1. 扩展配置层，支持 `VIDEO_ROOT_n` 与 `APP_PASSWORD`
2. 调整 ORM 模型与 SQLite 初始化结构，补充 `camera_no`
3. 重构索引任务，按通道扫描与按通道重建
4. 改造查询接口，统一增加 `camera` 维度
5. 前端增加通道切换与登录页
6. 为所有业务接口加认证守卫
7. 补充后端与前端测试，确保第一阶段能力不回退

## 13. 最终结论

第二阶段继续保持“本地工具优先”的产品定位，不引入摄像头管理和用户管理，而是通过极少量配置项实现多摄像头和单密码登录。

本方案的关键设计点只有三个：

- 摄像头通过 `VIDEO_ROOT_n` 配置，并统一映射为 `camera_no`
- 所有索引与查询都按 `camera_no + day` 维度执行
- 登录只校验 `.env` 中的 `APP_PASSWORD`，并通过 HttpOnly Cookie 维持登录态

在这一前提下，现有单路回放架构可以被平滑扩展，而不需要推倒重写。
